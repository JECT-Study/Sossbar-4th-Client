# 코드 품질 검증 절차

## 검증 순서

### 1단계: 정적 분석 (자동)

```bash
pnpm lint          # ESLint 오류 확인
pnpm build         # TypeScript 타입 에러 + 빌드 오류 확인
```

lint/build가 PASS여야 다음 단계로 진행한다.

### 2단계: 경계면 교차 비교 (수동)

파일을 하나씩 읽는 것이 아니라 **경계를 넘나드는 지점**을 집중 검사한다:

| 경계면                    | 검사 항목                                            |
| ------------------------- | ---------------------------------------------------- |
| API 함수 ↔ Zod 스키마     | 응답 shape와 스키마가 일치하는가                     |
| Zod 스키마 ↔ 훅 리턴 타입 | infer된 타입이 훅에서 올바르게 흐르는가              |
| 훅 ↔ 컴포넌트 props       | 훅 리턴값이 컴포넌트가 기대하는 타입과 맞는가        |
| 도메인 ↔ shared           | shared에 올린 것이 실제 2개 이상 도메인에서 쓰이는가 |

### 3단계: 컨벤션 체크리스트

```
[ ] default export: Next.js page/layout 외에 없는가
[ ] 화살표 함수 컴포넌트만 사용했는가
[ ] import type 분리했는가
[ ] cn() 사용 (clsx/twMerge 직접 사용 없는가)
[ ] any 없는가
[ ] console 없는가
[ ] 파일이 올바른 도메인/레이어에 위치하는가
[ ] index.ts에서 외부 공개 심볼만 re-export 하는가
```

### 4단계: 수정

발견된 문제는 직접 수정한다. 수정 후 `pnpm lint && pnpm build` 재실행으로 확인.

## 에스컬레이션 기준

직접 수정하지 않고 오케스트레이터에게 에스컬레이션:

- 도메인 배치 자체가 잘못된 경우 (planner 재실행 필요)
- 순환 참조 발생
- 기존 API 계약과 충돌하는 타입 불일치

## report 작성 기준

- "없음"도 명시 (미해결 문제 없음 = 신뢰도 높음)
- 수정한 내용은 파일명 + 무엇을 왜 바꿨는지 기록
- 최종 상태: `DONE` (정상) 또는 `ESCALATE` (에스컬레이션 필요)
