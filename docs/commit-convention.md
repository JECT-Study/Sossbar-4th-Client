## 커밋 컨벤션 개요

이 문서는 `commitlint.config.mjs`를 기준으로, 이 프로젝트에서 사용하는 **커밋 메시지 규칙과 그 이유**를 정리합니다.  
목표는 “**Git 로그만 읽어도 프로젝트의 히스토리와 맥락을 이해할 수 있게 만드는 것**”입니다.

### 큰 흐름 (Big Picture)

코드 변경이 커밋 → PR → 배포로 이어지는 과정에서 커밋 메시지는 다음 역할을 합니다.

```text
코드 변경
  │
  ▼
git add
  │
  ▼
커밋 메시지 작성
  │   (Conventional Commits 형식)
  ▼
Husky commit-msg 훅
  │   └─ pnpm dlx commitlint --edit $1
  │       - 타입/길이/형식 검사
  ▼
Git 로그 / PR 타임라인 / 릴리스 노트
  │
  ▼
버전 태깅(main 브랜치)
```

이 규칙 덕분에:

- “이 커밋이 **무엇을** 바꾸는지(type + subject)”
- “어느 범주의 변경인지(기능/버그/문서/디자인 등)”

를 누구나 빠르게 읽어낼 수 있습니다.

---

## 1. 기본 포맷

커밋 메시지는 **Conventional Commits** 형식을 따릅니다.

```text
<type>: <subject>
```

- `<type>`: 아래에 정의된 타입 중 하나를 사용합니다.
- `<subject>`:
  - 한글/영문 자유
  - **끝에 마침표(`.`)를 붙이지 않습니다.**
  - 한글은 평서형, 영문은 소문자 시작을 권장

예:

- `feat: 로그인 페이지 추가`
- `fix: 잘못된 리디렉션 경로 수정`

---

## 2. 허용 타입 (`type-enum`)

`commitlint.config.mjs`에서 허용하는 타입은 다음과 같습니다.

- **`feat`**: 새로운 기능 추가
  - 예: `feat: 프로젝트 생성 페이지 추가`
- **`fix`**: 버그 수정
  - 예: `fix: 잘못된 비밀번호 에러 메시지 수정`
- **`docs`**: 문서 수정/보완 (README, 설계 문서, 컨벤션 문서 등)
  - 예: `docs: eslint 컨벤션 문서 추가`
- **`style`**: 코드 스타일 변경 (로직 변경 없는 포맷팅, 세미콜론, 인덴트 등)
  - 예: `style: prettier 포맷 반영`
- **`refactor`**: 리팩터링 (동작은 동일, 구조/가독성 개선)
  - 예: `refactor: 유저 조회 로직 분리`
- **`perf`**: 성능 개선
  - 예: `perf: 리스트 렌더링 최적화`
- **`test`**: 테스트 코드 추가/수정/삭제
  - 예: `test: 로그인 훅 유닛 테스트 추가`
- **`chore`**: 빌드, 설정, 패키지 업데이트 등 기타 작업
  - 예: `chore: eslint 패키지 버전 업데이트`
- **`revert`**: 이전 커밋 되돌리기
  - 예: `revert: feat: 로그인 페이지 추가`
- **`ci`**: CI 설정 및 파이프라인 관련 변경
  - 예: `ci: main 브랜치에만 빌드 수행하도록 변경`
- **`design`**: UI/스타일 변경(기능 로직 변경 없이 디자인만 변경할 때)
  - 예: `design: 버튼 색상 및 타이포그래피 조정`

위 외의 타입(예: `build`, `release`, `hotfix` 등)은 기본 설정상 허용되지 않습니다.  
새로운 타입이 필요하면 **팀 합의 후 `commitlint.config.mjs`에 추가**해야 합니다.

## 3. 길이 및 형식 규칙

- **제목(헤더) 최대 길이**
  - `header-max-length: 72`
  - 커밋 메시지의 첫 줄(타입 + subject)을 **72자 이내**로 작성합니다.
  - 이유:
    - Git CLI, GitHub UI, 터미널 등에서 한 줄로 잘리지 않고 읽기 쉽습니다.

- **subject 끝에 마침표 금지**
  - `subject-full-stop: [2, 'never', '.']`
  - 예:
    - ❌ `feat: 로그인 페이지 추가.`
    - ✅ `feat: 로그인 페이지 추가`
  - 이유:
    - 로그를 나열해서 읽을 때 시각적 노이즈를 줄입니다.

- **subject 대소문자 규칙**
  - `subject-case`는 비활성화(`0`) 되어 있어, 한글/영문/대소문자에 대한 강제 규칙은 없습니다.
  - 권장:
    - 한글: “로그인 페이지 추가”, “버튼 스타일 정리” 같은 평서형
    - 영문: 소문자 시작 (`add login page`, `fix redirect bug`)

## 4. Commitlint & Husky 동작 방식

### 4-1. commit-msg 훅

`.husky/commit-msg`:

```bash
pnpm dlx commitlint --edit $1
```

- 커밋이 실행되면 Husky가 `commitlint`를 호출합니다.
- 현재 커밋 메시지가:
  - 허용된 `type`을 사용했는지
  - 길이/마침표 규칙을 지키는지
  - (추가된 룰이 있다면 그 룰들도 포함해) **자동으로 검사**합니다.

규칙을 어기면:

- 커밋이 거부되고,
- 어떤 규칙을 위반했는지 에러 메시지로 안내됩니다.

### 4-2. 왜 이렇게까지 하나?

- PR 목록, Git 로그를 빠르게 훑기만 해도:
  - “이 릴리즈에 어떤 기능이 들어갔는지(feat)”
  - “어떤 버그를 고쳤는지(fix/hotfix)”
  - “어떤 설정/빌드 변경이 있었는지(chore/ci)”
    를 알 수 있습니다.
- 릴리스 노트/변경 이력을 만들 때, 타입별로 **자동 그룹화**하기 쉬워집니다.

## 5. 좋은/나쁜 예시

- ✅ **좋은 예시**
  - `feat: 대시보드 차트 위젯 추가`
  - `fix: 다크 모드 전환 시 깜빡임 현상 수정`
  - `refactor: 프로젝트 카드 컴포넌트 분리`
  - `docs: eslint 컨벤션 문서 추가`
  - `design: 메인 페이지 타이포그래피 개선`

- ❌ **나쁜 예시**
  - `update: stuff` → `type`이 허용 목록에 없음
  - `feat(login): 페이지 추가` → 현재 포맷은 `feat: ...` 만 허용
  - `fix: 로그인 에러 수정.` → subject 끝에 `.` 사용
  - `feat: 이 커밋 메시지는 72자를 훌쩍 넘어서 굉장히 길게 작성된 예시로, 규칙을 위반합니다` → 72자 초과

## 6. 스스로 점검해볼 질문 (Why 5)

1. 이 커밋 메시지만 읽었을 때, **코드를 열지 않아도** 어떤 변경인지 상상할 수 있는가?
2. 이 변경은 정말 `feat`/`fix`/`refactor` 등 **지금 선택한 타입이 맞는가**, 아니면 다른 타입으로 쪼개는 게 더 명확한가?
3. 팀원이 Git 로그를 볼 때, 이 커밋이 **어떤 릴리즈 노트 항목** 아래에 들어가야 할지 바로 떠오르는가?
4. subject를 72자 이내로 줄이면서도, **핵심을 잃지 않고 더 간결하게** 표현할 수는 없는가?
5. 하나의 커밋에 너무 많은 변경이 섞여 있어서, 타입을 하나만 고르기 애매한 건 아닌가? 그렇다면 커밋을 나누는 것이 더 낫지 않은가?
